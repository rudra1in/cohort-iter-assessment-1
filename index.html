Q1

Transformer Architecture for LLMs

Scenario: Deploying a decoder-only Transformer (Llama 3, Mistral) for 64k–128k token context windows.

Task: Compare GQA vs MQA vs MHA. Why is GQA dominant for long contexts? Trade-offs: KV-cache memory, inference latency, output quality?
💡 Show Hint
Key points:
  • KV-cache: Grows O(seq_len). At 128k tokens, dominates memory.
  • GQA: Fewer KV heads, ~2× memory reduction vs MHA, minimal quality loss.
  • MQA: Single KV head, aggressive, often hurts quality.
  • FlashAttention-2/3: GQA benefits more from memory efficiency.

Q2

Prompt Engineering - Chain-of-Thought & Variants

Scenario: Building an AI-powered code review assistant for architectural decisions.

Task: Compare CoT, Tree-of-Thought (ToT), Self-Consistency (SC). When use each? Latency and cost implications for production?
💡 Show Hint
Key points:
  • Standard CoT: Single reasoning path, 1× latency.
  • Tree-of-Thought: Multiple branches, 3–5× latency, best for complex decisions.
  • Self-Consistency: Multiple runs + voting, reduces hallucination.

Q3

Prompt Engineering - Structured Output & Few-Shot

Scenario: Customer support AI extracting entities from support tickets → structured JSON.

Task: Compare zero-shot, few-shot, dynamic few-shot prompting. How prevent hallucinations? Enforce schema compliance? Cost vs accuracy?
💡 Show Hint
Key points:
  • Zero-shot: Pure schema, error-prone.
  • Few-shot: 2–3 examples, ~500–1000 extra tokens, much better consistency.
  • Dynamic few-shot: Select similar examples, higher accuracy but adds latency.
  • Schema enforcement: GBNF grammar (Llama), JSON mode (OpenAI).

Q4

LangChain: Document Loaders, Text Splitters & Runnables

Scenario: Building RAG system processing PDFs, HTML, CSVs for vector embeddings.

Task: Explain Document Loader + Text Splitter pipeline. Compare recursive vs sliding-window chunking. How does chunk size/overlap affect retrieval? Show Runnable pattern code.
💡 Show Hint
Key points:
  • Document Loaders: Abstract format differences (PDF, HTML, web).
  • Recursive: Respects hierarchy [\n\n, \n, ., " "], preserves structure.
  • Sliding window: Fixed size + overlap, simpler, good for dense text.
  • Chunk size: 300–800 tokens optimal (too small = fragmented, too large = bloat).

AWS EC2 & IAM for LLM Deployment

Scenario: Deploying Llama 70B (4-bit) on AWS EC2. Needs S3 access, SageMaker calls, CloudWatch logging.

Task: Design EC2 setup (instance type, storage). Define least-privilege IAM role (S3, SageMaker, CloudWatch). How isolate with Security Groups + VPC? Security vs convenience trade-offs?
💡 Show Hint
Key points:
  • Instance type: GPU compute (g5, p4), need 80GB+ memory for 70B model.
  • Storage: EBS gp3 for models, NVMe instance store for cache (faster but ephemeral).
  • IAM: Specific S3 bucket ARN, SageMaker endpoint ARN, CloudWatch logs (no wildcards).
  • Security Group: Inbound SSH (restricted), API 8000 (app subnet only), egress HTTPS 443.
