Transformer Architecture for LLMs

Scenario: You are deploying a decoder-only Transformer (like Llama 3, Mistral) for production long-form technical content generation with 64k–128k token context windows.

Task: Compare GQA (Grouped Query Attention) vs MQA (Multi-Query Attention) vs standard MHA (Multi-Head Attention). Explain why GQA is dominant in modern models for long contexts. What are the trade-offs in KV-cache memory, inference latency, and output quality?
💡 Show Hint
Consider:
  • KV-cache size: Grows O(seq_len) for each layer. At 128k tokens, this dominates memory.
  • GQA (Llama 3, Mistral): Fewer KV heads (e.g., 8 heads for queries, 1 for KV). ~2× memory reduction vs MHA. Minimal quality loss.
  • MQA (early experiments): Single KV head. Even more aggressive. Often hurts quality noticeably.
  • Hardware efficiency: FlashAttention-2/3 favors standard MHA; GQA benefits more from lower memory pressure.
  • Speculative decoding: Works better with fewer KV heads (faster draft model). This is a production win.
Structure your answer: Define each attention variant → Explain KV-cache memory math → Trade-off table (memory, latency, quality) → Recommend for long-context use cases.

Prompt Engineering - Chain-of-Thought & CoT Variants

Scenario: Your team is building an AI-powered code review assistant. The system must analyze complex architectural decisions and provide structured feedback.

Task: Compare three modern prompt engineering techniques: (1) Standard Chain-of-Thought (CoT), (2) Tree-of-Thought (ToT) for branching reasoning, (3) Self-Consistency (SC) with majority voting. When would you use each? What are latency and cost implications for production?
💡 Show Hint
Consider:
  • Standard CoT: "Let's think step-by-step" → single reasoning path. Works well for straightforward tasks. Latency: 1× baseline.
  • Tree-of-Thought (ToT): Explore multiple reasoning branches, prune low-confidence paths, backtrack. Best for complex decisions. Latency: 3–5× (multiple parallel calls).
  • Self-Consistency (SC): Run same prompt N times, aggregate results (voting/averaging). Reduces hallucination. Latency: N× baseline.
  • Current trend: Combining CoT + SC (multiple sampling with voting) is emerging as sweet spot for reasoning tasks.
  • Production use case: Code review is high-stakes (impact quality, time). Latency < 30s is typical requirement.
Structure your answer: Define each technique → Compare latency/cost/accuracy → Build a decision matrix → Recommend for code review use case.

Q3

Prompt Engineering - Structured Output & Few-Shot Learning

Scenario: Your company is building a customer support AI that needs to extract entities from support tickets and return structured JSON for downstream processing. The system must work with varying ticket formats and quality.

Task: Explain how to use few-shot prompting with structured output to improve consistency. Compare: (1) Zero-shot prompting with schema specification, (2) Few-shot prompting with 2–3 examples, (3) Dynamic few-shot selection. How do you prevent hallucinations and enforce schema compliance? What's the trade-off between accuracy and inference cost?
💡 Show Hint
Consider:
  • Zero-shot with schema: "Return JSON with fields: {customer_id, issue_type, urgency}". Simple but error-prone. LLM may hallucinate fields or wrong format.
  • Few-shot prompting: Include 2–3 real examples of input→output. Dramatically improves consistency. LLM learns pattern from examples. Adds ~500–1000 tokens per request.
  • Dynamic few-shot: Select most similar examples from a pool based on current ticket. Better than fixed examples for diverse inputs. Requires retrieval (latency cost).
  • Schema enforcement: Some models support constrained generation (e.g., Llama with GBNF grammar, OpenAI JSON mode). Prevents invalid JSON entirely.
  • In-context learning: LLMs learn from examples without parameter updates. This is the foundation of prompt engineering at scale.
Structure your answer: Define each approach → Compare accuracy/cost/latency → Discuss hallucination mitigation → Recommend for customer support use case.

LangChain: Document Loaders, Text Splitters & Runnables

Scenario: Your team is building a system that processes diverse documents (PDFs, HTML, CSVs) and needs to create embeddings for a vector database.

Task: Explain LangChain's Document Loader + Text Splitter pipeline. Compare recursive vs sliding-window chunking. How does chunk size/overlap affect RAG retrieval quality? Show a code pattern using Runnable and RunnablePassthrough.
💡 Show Hint
Consider:
  • Document Loaders: PDF/markdown/web loaders extract text. LangChain abstracts format differences.
  • Recursive chunking: Split on [\n\n, \n, . , " "] hierarchy. Preserves logical structure. Better for documents with clear hierarchy.
  • Sliding window: Fixed chunk size with overlap (e.g., 200 tokens, 50 overlap). Simpler. Good for dense text without structure.
  • Chunk size impact: Too small (50 tokens) = fragmented context, low quality retrieval. Too large (2000 tokens) = LLM context bloat, slower inference. Sweet spot: 300–800 tokens.
  • Runnable: LangChain's composable unit. RunnablePassthrough passes input through unchanged. Allows piping: loader | splitter | embedder.
Structure your answer: Explain loaders & splitters → Compare chunking strategies → Show code pattern → Discuss optimal chunk sizing for your use case.

AWS EC2 & IAM for LLM Deployment

Scenario: You're deploying a Llama 3 70B model (4-bit quantized) on AWS EC2 for inference. The server needs to access S3 for model weights, call SageMaker for batch jobs, and stream logs to CloudWatch.

Task: Design an EC2 instance setup (instance type, storage, networking). Define an IAM role with least-privilege permissions for S3, SageMaker, and CloudWatch. Explain how you'd use EC2 Security Groups + VPC to isolate the inference server. What are the security vs. convenience trade-offs?
💡 Show Hint
Consider:
  • Instance type: GPU compute (g4dn, g5 for NVIDIA/AMD GPUs). Llama 70B (4-bit) ≈ 35GB → need ≥ 80GB GPU memory → g5.24xlarge or p4 instances.
  • Storage: EBS gp3 for OS/app (50GB), separate high-throughput EBS for model cache, or local NVMe instance store (faster but ephemeral).
  • IAM role policy: Attach role to EC2 instance. Policies: S3 (GetObject on model bucket), SageMaker (InvokeEndpoint, Describe*), CloudWatch (PutMetricData, Logs:CreateLogGroup/CreateLogStream/PutLogEvents).
  • Security Group: Inbound: SSH (22, restricted to admin IPs), API (8000, restricted to app servers only). Egress: S3, SageMaker, CloudWatch (HTTPS 443).
  • Least privilege: Avoid wildcards (*). Specific S3 bucket ARN, specific SageMaker endpoint ARN. Document why each permission exists.
Structure your answer: Instance sizing → Storage design → IAM policy JSON → Security Group rules → Justify trade-offs.
