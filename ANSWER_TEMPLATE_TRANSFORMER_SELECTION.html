The Scenario

You are an AI Engineer at a fast-growing company. Your team needs to select and deploy a decoder-only Transformer generative model for a production application that generates long-form technical content (documentation, reports, code explanations) with context windows up to 64k–128k tokens. The model must balance quality, cost, latency, and maintainability.

Model families in scope: Llama 3/4, Mistral, Gemma, Qwen, Phi-3/4, Grok, Claude, GPT variants — or any other real model you know.
Your task: Recommend the most suitable model(s) and justify your choices across architecture, usability, and production readiness.
Part A Model Selection Criteria ~25 marks
A1
What key Transformer architecture attributes would you evaluate when selecting a model for this use case? Address: attention mechanism, positional encoding, context length handling, and architecture modifications like GQA, SwiGLU, RMSNorm.
8 marksArchitecture deep-dive
What to cover — attribute by attribute:
  • Attention mechanism — Standard MHA vs GQA (Grouped Query Attention) vs MQA. GQA reduces KV-cache memory at long contexts. Used in Llama 3, Mistral. Why does this matter at 64k+ tokens?
  • Positional encoding — Compare RoPE (Rotary Position Embedding, scales well, most modern models), ALiBi (attention bias, extrapolates), YaRN (RoPE extension for context stretching). For 128k context, RoPE + YaRN is the dominant choice.
  • Context length handling — KV-cache size grows linearly with sequence length. Sliding window attention (Mistral) vs full attention. FlashAttention-2/3 for memory efficiency.
  • Norm + activation — RMSNorm (faster than LayerNorm, used in Llama/Mistral), SwiGLU activation (better than ReLU/GELU for quality, ~8% more compute). These affect per-token inference speed.
Suggested answer structure: 1) List each attribute as a sub-heading → 2) Define it briefly → 3) State why it matters for this specific use case (128k context, technical content) → 4) Name a model that exemplifies it.

A3
Which model characteristics matter most for technical content generation (documentation, code explanations, reports) versus creative writing? Where do the priorities diverge?
5 marksDomain reasoning
Key dimensions to compare:
  • Factual accuracy — Technical content demands low hallucination. Creative writing tolerates (even rewards) invention. Hint: How does training data composition affect this?
  • Code understanding — Models trained with large code corpora (Llama 3, Qwen, DeepSeek) handle code explanations better. Pure language models may generate plausible-sounding but wrong code.
  • Structural coherence over long output — Technical docs must remain consistent across 10k+ tokens. Temperature must be lower. Models with stronger instruction following matter more here.
  • Diversity vs precision — Creative writing benefits from high temperature, diverse sampling. Technical content: low temperature, top_p tighter, possibly beam search / greedy.
  • Format adherence — Technical content often needs Markdown, structured headers, consistent code blocks. RLHF/instruction-tuned models handle this better.
Frame your answer as two columns: "For technical content, X matters because..." vs "For creative writing, Y matters because..." Then pick one model suited to each.

Part B Usability & Production Readiness ~15 marks
B1
How would you evaluate inference usability factors: quantization compatibility (INT4, INT8, FP8) and speculative decoding support? What are the tradeoffs for each?
7 marksInference engineering
Quantization — key facts to include:
  • INT8 (8-bit) — ~2x memory reduction vs FP16, minimal quality loss on most tasks, widely supported (bitsandbytes, GPTQ, vLLM). Good default for production.
  • INT4 (4-bit) — ~4x memory reduction. Quality drops noticeably on complex reasoning. Use GPTQ or GGUF (llama.cpp). Best for local/edge deployment or fast prototyping.
  • FP8 — Hardware-native (H100, upcoming GPUs). Minimal quality loss vs INT8. vLLM supports FP8. The right choice on modern infrastructure.
  • Trade-off pattern: Memory ↓ → Throughput ↑ → Quality ↓
Speculative decoding:
  • A small "draft" model generates candidate tokens; the large model verifies in parallel. Reduces latency 2–3x for greedy/low-temperature outputs (technical content fits this perfectly!).
  • Requires a compatible draft model — e.g. Llama-3-8B drafting for Llama-3-70B. Mistral supports this natively in vLLM.
  • Hint: Why does speculative decoding work especially well for documentation generation? (Hint: predictable, structured output)
Answer structure: 1) Explain each quantization level with its memory/quality tradeoff → 2) State which you'd choose for this production use case and why → 3) Explain speculative decoding mechanism → 4) Argue why it fits technical content generation.

B2
Explain the trade-offs between using a larger dense model (e.g. Llama-3-70B) vs a smaller Mixture-of-Experts (MoE) model (e.g. Mixtral 8x7B, Qwen-MoE) for this scenario.
8 marksArchitecture tradeoff
MoE mechanics to explain:
  • MoE activates only a subset of parameters per token (e.g. Mixtral 8x7B has 47B total params but activates only ~13B per token → cheaper per-token compute, expensive memory)
  • Dense model advantage: simpler deployment, all parameters always active, predictable memory, better support for quantization and speculative decoding
  • MoE advantage: higher parameter count at same compute budget → often better quality at equivalent FLOPS. Routing diversity may help technical content (different experts for code vs prose)
  • MoE problem for this use case: All experts must stay in GPU memory. Mixtral 8x7B needs ~90GB for FP16 → 2×A100s. Vs Llama-3-70B at ~140GB (4-bit: ~35GB). Memory pressure is higher for MoE at long contexts (KV-cache also grows).
Decision framework to argue:
If you have multi-GPU setup with high memory → MoE can give better quality/cost ratio.
If you have single A100 or inference budget constraint → Dense + INT8/INT4 is more predictable.
For 128k technical content at scale → Dense model with GQA + FlashAttention may be more operationally stable.

If cost becomes a major constraint, how would you decide between: running a high-quality open-source model locally/on-prem vs using a hosted frontier model via API (Claude, GPT-4, Gemini)?
+5 bonus marksBusiness + engineering reasoning
Decision factors to structure around:
  • Volume economics — At low volume (< 1M tokens/day): API is cheaper (no GPU capex). At high volume (>10M tokens/day): on-prem quickly wins on per-token cost. Build a break-even model.
  • Data privacy / compliance — If generating internal technical docs with proprietary code → data leaving to an API is a risk. On-prem required for regulated industries (banking, health, defence).
  • Quality ceiling — GPT-4o, Claude Opus still outperform most open-source on complex multi-step reasoning. If output quality directly impacts revenue/safety → the quality premium may justify API cost.
  • Latency + SLA — API latency is unpredictable under load. On-prem gives deterministic SLA. For batch documentation generation, API is fine. For real-time code assist, on-prem wins.
  • Operational complexity — On-prem = model updates, infra maintenance, GPU procurement, vLLM/TGI ops team. API = just a key rotation. Factor this into TCO.
Frame as a decision matrix: Show 4–5 factors, rate each option, then give a concrete recommendation tied to the scenario (technical content, 128k context, mid-sized company).
